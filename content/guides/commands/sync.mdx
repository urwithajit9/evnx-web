`evnx sync` maintains consistency between your local environment file (`.env`) and your team template file (`.env.example`). It supports bidirectional synchronization with intelligent placeholder generation, security guards, and automation-friendly flags.

<Prerequisites>
  <Prerequisite href="/guides/getting-started/installation">Install evnx</Prerequisite>
  <Prerequisite href="/guides/commands/add">Add environment variables</Prerequisite>
</Prerequisites>

## Command signature

```bash
evnx sync [OPTIONS]
```

**Options:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--direction`, `-d` | `enum` | `forward` | Sync direction: `forward` or `reverse` |
| `--placeholder`, `-p` | `bool` | `false` | Force placeholder values when adding variables |
| `--dry-run`, `-n` | `bool` | `false` | Preview changes without writing files |
| `--force`, `-f` | `bool` | `false` | Skip interactive prompts (CI/CD mode) |
| `--template-config` | `string` | — | Path to custom placeholder JSON config |
| `--naming-policy` | `enum` | `warn` | Handle non-standard names: `warn`, `error`, `ignore` |
| `--verbose`, `-v` | `bool` | `false` | Enable detailed debug output |
| `--help`, `-h` | `bool` | `false` | Display help information |

**Environment Variables:**

| Variable | Values | Default | Description |
|----------|--------|---------|-------------|
| `NO_COLOR` | `1`, `true` | `false` | Disable colored output for terminals |
| `CI` | `1`, `true` | `false` | Auto-enable non-interactive mode |

## Sync directions

### Forward sync (`.env` → `.env.example`)

**Purpose:** Update the team template with new variables from your local environment.

```bash
evnx sync --direction forward
# or simply:
evnx sync  # forward is default
```

<Callout type="info" title="When to use forward sync">
Use forward sync when you've added new environment variables to your local `.env` file and want to share them with your team by updating `.env.example`.
</Callout>

**Flow:**
```
1. Parse .env and .env.example
2. Find variables in .env missing from .env.example  
3. For each missing variable:
   • Generate smart placeholder (based on name patterns)
   • OR use actual value (with explicit confirmation)
4. Append to .env.example with sync comment
5. Optionally add description comment
```

**Interactive options:**
```
? Add these to .env.example?
❯ Yes, add with placeholder values (recommended)
  Yes, add with actual values ⚠️ SECURITY RISK
  Let me choose individually
  No, skip
```

<Callout type="danger" title="Security warning">
Selecting "add with actual values" can expose secrets if `.env.example` is committed to version control. This option requires **double confirmation** and logs an audit message in verbose mode.
</Callout>

---

### Reverse sync (`.env.example` → `.env`)

**Purpose:** Update your local environment with new variables from the team template.

```bash
evnx sync --direction reverse
```

<Callout type="info" title="When to use reverse sync">
Use reverse sync when your team has updated `.env.example` with new required variables, and you need to add them to your local `.env` file.
</Callout>

**Flow:**
```
1. Parse .env.example and .env
2. Find variables in .env.example missing from .env
3. For each missing variable:
   • Add with placeholder value, OR
   • Prompt for real value interactively
4. Append to .env with sync comment
5. Remind to replace placeholders with real credentials
```

**Interactive options:**
```
? Add to .env?
❯ Yes, with placeholder values
  Yes, prompt me for real values
  Let me choose individually
  No, skip
```

## Smart placeholder generation

evnx automatically generates safe placeholder values based on variable name patterns.

### Built-in rules

| Variable name contains | Generated placeholder |
|----------------------|---------------------|
| `SECRET`, `KEY`, `TOKEN` | `YOUR_KEY_HERE` |
| `PASSWORD`, `PASS` | `YOUR_PASSWORD_HERE` |
| `URL` (postgresql://) | `postgresql://user:password@localhost:5432/dbname` |
| `URL` (redis://) | `redis://localhost:6379/0` |
| `URL` (http/https) | `https://your-api-url-here.com` |
| `PORT` | `8000` |
| `DEBUG` | `true` |
| `HOST`, `SERVER` | `localhost` |
| Default | `YOUR_VALUE_HERE` |

### Custom placeholder configuration

Create `.evnx/placeholders.json` for team-specific patterns:

```json
{
  "$schema": "https://evnx.dev/schemas/placeholders.v1.json",
  "description": "Team placeholder conventions",
  "patterns": {
    ".*_API_KEY$": "sk_demo_XXXXXXXXXXXXXXXX",
    ".*_SECRET$": "demo_secret_placeholder",
    "DATABASE_URL": "postgresql://demo:demo@localhost:5432/demo_db",
    "REDIS_URL": "redis://localhost:6379/0",
    ".*_PORT$": "8080",
    ".*_HOST$": "localhost",
    "DEBUG": "true",
    "ENVIRONMENT": "development"
  },
  "default": "REPLACE_IN_PRODUCTION",
  "allow_actual": ["APP_NAME", "APP_VERSION", "ENVIRONMENT"]
}
```

**Usage:**
```bash
evnx sync --template-config .evnx/placeholders.json
```

<Callout type="tip" title="Pattern matching">
Patterns use **regex** with case-insensitive matching. `.*_API_KEY$` matches `STRIPE_API_KEY`, `AWS_API_KEY`, etc.
</Callout>

## Workflows

### Workflow 1: New project without `init`

Start fresh by adding variables manually, then sync to create `.env`.

```bash
# Step 1: Add custom variables (creates .env.example)
evnx add custom

Variable name: DATABASE_URL
Example/placeholder: postgresql://localhost:5432/mydb
Add another? yes

Variable name: API_KEY  
Example/placeholder: YOUR_KEY_HERE
Add another? no

✓ Added 2 custom variables

# Step 2: Create .env from template
evnx sync --direction reverse

ℹ️  .env not found, creating from .env.example
✓ Created .env
⚠️  Replace placeholder values with real credentials!

# Step 3: Edit .env with real values
nano .env

# Step 4: Secure permissions
chmod 600 .env

# Step 5: Validate
evnx validate
✓ All variables present and valid
```

<Callout type="tip" title="No init required">
You don't need to run `evnx init` to use sync. The `add` command creates `.env.example` automatically, and `sync reverse` creates `.env`.
</Callout>

### Workflow 4: CI/CD validation

Automated checks to ensure `.env.example` stays up to date.

```yaml
# .github/workflows/env-sync-check.yml
name: Env Sync Check
on: [push, pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Install evnx
        run: cargo install evnx
        
      - name: Check sync state
        run: |
          evnx sync \
            --direction forward \
            --dry-run \
            --force \
            --naming-policy error \
            --template-config .evnx/placeholders.json
          
          if ! git diff --quiet .env.example; then
            echo "::error ::.env.example is out of sync!"
            echo "Run 'evnx sync --direction forward' locally and commit."
            exit 1
          fi
          
      - name: Check for secrets
        run: |
          if grep -iE "(sk_live|password|secret)" .env.example | \
             grep -v "YOUR_\|placeholder\|demo"; then
            echo "::error ::Potential secrets in .env.example!"
            exit 1
          fi
```

<Callout type="warning" title="Never commit .env">
Always ensure `.env` is in `.gitignore`. The sync command will warn if permissions are too permissive, but Git won't block you from accidentally committing secrets.
</Callout>

## Naming convention validation

Enforce consistent environment variable naming across your team.

### Policy options

| Policy | Behavior | Best for |
|--------|----------|----------|
| `warn` | Shows warning, continues | Local development (default) |
| `error` | Fails on non-standard names | CI/CD validation |
| `ignore` | No validation | Legacy projects |

### Standard convention: `SCREAMING_SNAKE_CASE`

```
✅ DATABASE_URL, API_SECRET_KEY, DEBUG_MODE, MY_APP_PREFIX_VAR
❌ databaseUrl, apiKey, debug-mode, myAppVar
```

**Examples:**
```bash
# Warn but continue (default)
evnx sync --naming-policy warn
⚠️  Variable 'myVar' doesn't follow convention (UPPERCASE_WITH_UNDERSCORES)

# Fail in CI on non-standard names
evnx sync --naming-policy error --force
✗ Non-standard variable name 'myVar'
# Exit code: 1

# Ignore conventions for legacy projects
evnx sync --naming-policy ignore
# No warnings, proceeds silently
```

## Security features

### Protection layers

| Layer | Description |
|-------|-------------|
| Default to placeholders | Never actual values unless explicitly chosen |
| Double confirmation | Two prompts before adding actual values to templates |
| Audit logging | Verbose mode logs sensitive operations |
| Permission checks | Warns and auto-sets `0o600` on `.env` (Unix) |
| Atomic writes | Prevents file corruption using temp files + rename |
| Dry-run mode | Preview before applying changes |

### Actual values warning flow

When selecting "add with actual values":

```
┌─ ⚠️  SECURITY WARNING ─────────────────────────┐
│ Adding actual values to .env.example may       │
│ expose secrets if this file is committed to    │
│ version control.                                │
│                                                 │
│ Only proceed if you are certain .env.example   │
│ is:                                             │
│ • Gitignored, OR                               │
│ • Contains no sensitive data                   │
└────────────────────────────────────────────────┘

? Are you absolutely sure you want to add ACTUAL VALUES? (y/N)
❯ n

ℹ️  Aborted: values not added
```

### File permissions

```bash
# Check current permissions
ls -la .env
-rw------- 1 user user 256 Mar 3 10:00 .env  # ✓ Correct (600)
-rw-r--r-- 1 user user 256 Mar 3 10:00 .env  # ⚠️ Too permissive (644)

# Set correct permissions
chmod 600 .env

# evnx sync auto-sets on write (Unix)
evnx sync --direction reverse
✓ Created .env (permissions: 600)
```

## Exit codes

| Code | Meaning | Use Case |
|------|---------|----------|
| `0` | ✅ Sync completed successfully | Success in scripts/CI |
| `1` | 🚨 Sync failed (missing files, validation errors) | Fail pipeline on issues |
| `2+` | ⚠️ Runtime errors (IO, parsing failures) | Debug tooling issues |

<Callout type="tip" title="Testing exit codes">
```bash
# Test success case
evnx sync --dry-run
echo $?  # Should be 0

# Test failure case (missing .env)
rm .env
evnx sync
echo $?  # Should be 1
```
</Callout>

## Troubleshooting

### ❌ "File not found: .env"

```
✗ File not found: .env

┌─ 💡 Getting Started ─────────────────────────┐
│ It looks like this project hasn't been       │
│ initialized yet.                              │
│                                               │
│ To create .env and .env.example files, run:  │
│                                               │
│ $ evnx init                                   │
│                                               │
│ Or if you have an existing .env file, place  │
│ it in this directory and try again.           │
└──────────────────────────────────────────────┘
```

**Solutions:**
1. Run `evnx init` to generate files from a blueprint
2. Copy your existing `.env` to the project directory
3. Use `evnx add custom` to create variables manually, then `sync reverse`

---

### ❌ "File not found: .env.example"

```
✗ File not found: .env.example

┌─ 💡 Getting Started ─────────────────────────┐
│ The template file .env.example is missing.   │
│                                               │
│ To create it, either:                        │
│                                               │
│ 1. Run 'evnx init' to generate from template │
│ 2. Create .env.example manually              │
└──────────────────────────────────────────────┘
```

**Solutions:**
1. Run `evnx init --framework <your-framework>`
2. Create `.env.example` manually with variable names
3. Use `evnx add custom` which auto-creates `.env.example`

---

### ❌ Placeholder not matching custom pattern

```bash
# Verify JSON is valid
cat .evnx/placeholders.json | jq .

# Test pattern matching
evnx sync --template-config .evnx/placeholders.json --verbose

# Check regex syntax (patterns are case-insensitive)
# Pattern: .*_API_KEY$ matches STRIPE_API_KEY, AWS_API_KEY
```

---

### ❌ Sync not detecting changes

```bash
# Check file contents
cat .env
cat .env.example

# Verify parsing with verbose mode
evnx sync --verbose --dry-run

# Check for hidden characters
file .env .env.example
cat -A .env  # Shows hidden characters like ^M (Windows line endings)

# Fix line endings if needed
dos2unix .env .env.example  # Or: sed -i 's/\r$//' .env
```

# Ensure only your user account has Read/Write access
